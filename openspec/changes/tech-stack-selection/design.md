# Design: Tech Stack Selection & Project Scaffold

## Technical Approach

Bootstrap a greenfield Vite + React 18 project using vanilla JavaScript (no TypeScript). All
tooling — Tailwind CSS, shadcn/ui, Zustand, Vitest, ESLint v8, Prettier — is installed and
configured up-front so every subsequent change starts from a clean, consistent foundation. The
design covers only the scaffold layer; no application logic, music domain types, store shape, or
engine/audio architecture is included.

The scaffold is intentionally minimal: placeholder files hold directory slots, a single smoke test
validates the wiring, and a CI pipeline enforces lint → test → build on every push.

## Architecture Decisions

### Decision: Vite over CRA or Next.js

**Choice**: Vite 5 as the build tool and dev server.  
**Alternatives considered**: Create React App (deprecated), Next.js (SSR-first).  
**Rationale**: CRA is no longer maintained. Next.js server-side rendering conflicts with the Web
Audio API — `AudioContext` requires a browser environment and cannot be instantiated during
server-side rendering. Vite is framework-agnostic, fast, and purpose-built for browser-first SPAs.

### Decision: Vanilla JavaScript over TypeScript

**Choice**: Plain JS with JSDoc annotations for documentation.  
**Alternatives considered**: TypeScript strict mode.  
**Rationale**: Explicit project choice to keep the initial toolchain simple. JSDoc provides
editor intelligence and inline documentation without the compilation overhead or migration burden.
TypeScript migration is explicitly out of scope and deferred to a future change.

### Decision: Vitest co-located in vite.config.js

**Choice**: `test` block embedded directly in `vite.config.js`.  
**Alternatives considered**: Separate `vitest.config.js` file.  
**Rationale**: Vitest shares Vite's plugin pipeline and configuration by design. Co-location
avoids duplication of the `resolve.alias` and plugin settings, and eliminates a separate config
file from the project root.

### Decision: ESLint v8 (.eslintrc.cjs) over ESLint v9 (flat config)

**Choice**: ESLint v8 with `.eslintrc.cjs`.  
**Alternatives considered**: ESLint v9 with `eslint.config.js` (flat config).  
**Rationale**: The Vite scaffold generates an ESLint v9 flat config by default. However, flat
config support in editor plugins and many community presets is still maturing. The v9 file
generated by the scaffold was deleted and replaced with a `.eslintrc.cjs` using the stable v8
format. Migration to v9 flat config is a separate, intentional future change.

### Decision: @babel/preset-react added to vite.config.js

**Choice**: Pass `babel.presets: [['@babel/preset-react', { runtime: 'automatic' }]]` to
`@vitejs/plugin-react`.  
**Alternatives considered**: Rename test files to `.jsx`, use a separate Babel config file.  
**Rationale**: `@vitejs/plugin-react` v4 calls `canSkipBabel()` internally and skips the Babel
transform when it detects the file is "already transformed" (which happens in Vitest mode for
`.js` files). Adding the preset forces the JSX transform to run, allowing `smoke.test.js` (a
`.js` file) to contain JSX without errors. Renaming all test files to `.jsx` would work but
adds unnecessary friction.

### Decision: jsconfig.json for path alias IDE resolution

**Choice**: `jsconfig.json` with `compilerOptions.paths: { "@/*": ["src/*"] }`.  
**Alternatives considered**: No jsconfig (rely solely on Vite alias).  
**Rationale**: Vite's `resolve.alias` handles path resolution at build/dev time, but editors
(VS Code, WebStorm) rely on `jsconfig.json` / `tsconfig.json` to resolve non-relative imports
for IntelliSense, go-to-definition, and auto-import. Without `jsconfig.json`, `@/` imports show
as unresolved in the editor even though the build works correctly.

### Decision: Tailwind CSS v3 over v4

**Choice**: Tailwind CSS v3 (stable).  
**Alternatives considered**: Tailwind CSS v4 (alpha at time of scaffold).  
**Rationale**: Tailwind v4 was in alpha at project inception and is not production-ready. v3 is
the current stable release. Migration to v4 is deferred until it reaches GA.

### Decision: Zustand v5 over Context API or Redux

**Choice**: Zustand v5 for global state management.  
**Alternatives considered**: React Context + useReducer, Redux Toolkit.  
**Rationale**: Zustand requires zero provider wrapping, has minimal boilerplate, and supports
selector-based re-renders out of the box. Redux Toolkit adds significant ceremony for MVP scope.
The store is a placeholder at this stage; its shape is defined by future changes.

### Decision: Manual shadcn/ui initialization

**Choice**: Manually create `components.json`, `src/lib/utils.js`, and `src/components/ui/button.jsx`.  
**Alternatives considered**: `npx shadcn@latest init` CLI.  
**Rationale**: The shadcn/ui v4 CLI is fully interactive and cannot be run non-interactively in a
scripted or CI context. All three artifacts are simple enough to create by hand with the correct
content.

## Data Flow

This change establishes the scaffold only. There is no runtime data flow to describe. The
directory structure defines the *boundaries* where future data flows will be implemented:

```
src/main.jsx
    └── renders <App />
            └── (future) reads from src/store/index.js  [Zustand]
            └── (future) calls   src/engine/index.js    [sequencer]
            └── (future) calls   src/audio/index.js     [Tone.js]
            └── renders  src/components/                [UI layer]
```

The CI pipeline data flow is linear and fail-fast:

```
git push / PR → CI trigger
    → npm ci
    → npm run lint   (fail → stop)
    → npm run test   (fail → stop)
    → npm run build  (fail → stop)
```

## Interfaces / Contracts

### cn() utility — `src/lib/utils.js`

```js
import { clsx } from 'clsx';
import { twMerge } from 'tailwind-merge';

/**
 * Merge Tailwind class names, resolving conflicts.
 * @param {...(string|undefined|null|false)} inputs
 * @returns {string}
 */
export function cn(...inputs) {
  return twMerge(clsx(inputs));
}
```

This is the only public interface created by this change. All other modules are empty placeholders.

### Placeholder module contract

Each placeholder file (`engine/index.js`, `audio/index.js`, `store/index.js`, `types/index.js`,
`components/index.js`) exports nothing. They exist only to establish the directory structure.
Future changes own these modules entirely.

### vite.config.js test block

```js
test: {
  globals: true,            // describe/it/expect available without import
  environment: 'jsdom',     // DOM APIs available in tests
  setupFiles: ['./src/__tests__/setup.js'],
  coverage: { provider: 'v8', reportsDirectory: './coverage' },
}
```

### CI job contract

The CI job is fail-fast. Each step must exit 0 before the next runs:

```
checkout → setup-node (v20, cache: npm) → npm ci → lint → test → build
```

Triggers: `push` on all branches; `pull_request` targeting `main`.

## Testing Strategy

| Layer | What to Test | Approach |
|-------|-------------|----------|
| Smoke | React app renders without crashing | RTL `render(<App />)`, assert `container` in document |
| Smoke | DOM environment is available in tests | Assert `document.body` is defined |
| CI gate | All checks pass on every push | GitHub Actions workflow runs lint → test → build |

No unit, integration, or E2E tests are added in this change beyond the scaffold smoke test.
Application logic tests are the responsibility of the features that introduce that logic.

## Migration / Rollout

No migration required. This is a greenfield project — there is no prior state to migrate.

To roll back: delete the project directory. No external systems are affected.
